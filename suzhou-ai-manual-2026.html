全链路追溯功能操作手册

1. 手册概述

本手册是SCSAI数字员工平台全链路追溯功能的完整操作指南，旨在帮助用户全面掌握从需求提出到技能部署的全流程追溯能力。全链路追溯是平台的核心功能之一，它实现了需求→开发→测试→部署→运行→优化的完整闭环追踪，使每一个业务需求都可以快速定位到对应的代码实现、测试用例、部署版本和运行日志。

1.1 全链路追溯的核心价值

  • 端到端追踪：从需求提出到技能上线，全流程可追溯、可审计
  • 问题快速定位：当线上出现问题时，可快速定位到具体的代码变更、配置修改或测试遗漏
  • 知识图谱关联：自动建立需求、代码、测试、部署之间的关联关系，形成知识网络
  • 合规审计支持：满足金融级合规要求，提供完整的操作审计记录

1.2 追溯链条说明

🔗
全链路追溯链
需求ID → 需求拆解 → 技能设计 → 代码实现 → 测试用例 → 部署版本 → 运行实例 → 运行日志 → 优化反馈

每一个环节都通过唯一的追溯ID进行关联，形成完整的追踪链条。任意环节出现问题，都可以通过追溯链向上下游追溯问题根因。

2. 系统登录与主界面

2.1 登录步骤

1
打开登录页面

在浏览器中访问平台地址：https://scsai.example.com

2
输入账号密码

使用分配的工号和密码登录，首次登录需绑定企业微信或手机号

3
完成认证

根据配置的单点登录方式完成认证（企业微信扫码/短信验证码/密钥器）

4
进入工作台

认证成功后自动跳转到个人工作台

2.2 主界面介绍

登录成功后进入主界面，分为以下几个区域：

  • 顶部导航栏：包含工作台、需求中心、技能市场、追溯中心、部署管理、个人中心等模块入口
  • 左侧菜单：根据用户角色动态显示可访问的功能菜单
  • 主工作区：显示当前选中模块的核心内容
  • 右侧详情面板：用于显示选中项的详细信息或操作面板

2.3 工作台使用指南

📊
个人工作台

工作台是个人的工作 dashboard，包含以下核心模块：

  • 我的待办：显示需要处理的需求、待审批事项、待修复缺陷
  • 我的在进行：显示当前正在进行的工作项及进度
  • 我的完成：显示已完成的工作项统计
  • 团队动态：显示团队成员的最新操作
  • 快捷操作：常用功能的一键入口
💡 提示：工作台支持自定义布局，点击右上角「定制工作台」可调整模块位置和显示内容。

3. 需求管理模块

3.1 提交需求

点击「需求中心」→「新建需求」，填写需求表单：

字段说明必填
需求标题简要描述需求内容是
需求描述详细说明业务背景、目标、约束条件是
需求来源客户反馈/市场调研/竞品分析/内部优化是
优先级P0/P1/P2/P3（P0最高）是
期望上线时间期望完成日期否
关联需求关联的其他需求ID否
附件需求相关文档、原型、截图否
💡 提示：提交需求时会自动生成唯一的需求ID（如 REQ-2026-00123），该ID在整个追溯链中使用。

3.2 查看需求

在需求列表页面，可以按以下方式筛选和查看需求：

  • 全部需求：查看所有提交的需求
  • 我的需求：查看我提交的需求
  • 待处理：查看待产品经理确认的需求
  • 已通过：查看已通过评审的需求
  • 已驳回：查看被驳回的需求及原因

3.3 需求追溯

🔍
需求追溯功能

点击任意需求进入详情页，可查看「追溯」tab页，展示该需求的完整追溯链：

  • 需求信息：需求的基本信息和状态变更历史
  • 拆解记录：产品经理拆解的子需求列表
  • 技能关联：该需求关联的技能及其状态
  • 测试用例：关联的测试用例及执行结果
  • 部署记录：该需求的部署历史
  • 运行日志：上线后的运行日志摘要
⚠️ 注意：需求一旦开始开发，请勿直接修改需求内容，应通过「需求变更」流程进行更新，以便保留变更历史。

4. 技能开发模块

4.1 创建技能

点击「技能开发」→「新建技能」：

1
填写基本信息

技能名称、技能描述、所属分类、技能图标

2
关联需求

选择该技能对应的需求ID（可多选）

3
选择开发方式

可视化编排 / 代码开发 / 混合模式

4
配置技能参数

输入参数、输出参数、配置项、环境变量

4.2 可视化编排

使用可视化编排器，通过拖拽节点的方式构建技能逻辑：

  • 输入节点：配置技能触发方式和输入参数
  • 处理节点：LLM调用、数据处理、条件分支、循环等
  • 输出节点：配置技能返回结果
  • 工具节点：集成外部API、数据库查询、文件处理等
💡 提示：可视化编排支持实时预览和测试，可以在编排过程中随时点击「测试运行」验证逻辑。

4.3 代码管理

点击「代码编辑」进入代码开发界面：

  • 在线IDE：基于 VS Code 的在线代码编辑器
  • 代码仓库：每个技能对应一个 Git 仓库
  • 版本管理：Git 版本控制，支持分支和标签
  • 代码审查：提交 PR 后需经过 Code Review 才能合并

4.4 版本管理

📚
版本管理规范
  • 版本号格式：主版本.次版本.修订号（如 v1.2.3）
  • 主版本变更：不兼容的 API 变更
  • 次版本变更：向后兼容的功能新增
  • 修订号变更：向后兼容的问题修复
  • 每个版本：对应唯一的代码快照，可随时回滚

5. 测试管理模块

5.1 测试用例设计

点击「测试管理」→「新建测试用例」：

字段说明
用例名称描述测试用例的目的
关联技能选择被测试的技能
前置条件执行测试前的准备条件
输入数据测试输入参数
预期输出期望的返回结果
测试类型单元测试/集成测试/端到端测试

5.2 自动化测试

平台提供自动化测试能力：

  • 定时执行：配置 cron 表达式，自动执行测试
  • 触发执行：代码提交时自动触发测试
  • 部署前测试：部署前自动执行核心测试用例
  • 测试报告：自动生成测试报告，统计通过率
💡 提示：建议将测试用例覆盖率作为技能上线的门槛指标，建议核心功能测试覆盖率≥90%。

5.3 缺陷管理

在测试过程中发现的缺陷会自动记录到缺陷列表：

  • 缺陷等级：P0（ blocker）/P1（critical）/P2（major）/P3（minor）
  • 缺陷状态：新建/已确认/进行中/已修复/已验证/已关闭
  • 追溯关联：缺陷可追溯到对应的需求、技能、测试用例
  • 回归测试：修复后自动关联原测试用例进行验证

6.1 一键部署

当代码通过测试后，可发起部署申请：

1
选择部署版本

从版本列表选择需要部署的版本

2
选择环境

测试环境 / 预发布环境 / 生产环境

3
配置部署参数

副本数、资源配置、流量比例

4
提交部署

审批通过后自动执行部署

6.2 运行监控

部署完成后，在「运维管理」→「运行监控」查看：

  • 服务状态：运行实例数、健康状态、资源使用
  • 调用统计：QPS、响应时间、错误率
  • 资源监控：CPU、内存、磁盘、网络
  • 指标趋势：关键指标的时序图表

6.3 异常告警

⚡
告警规则配置
  • 告警类型：错误率阈值、响应超时、资源不足、部署失败
  • 告警级别：P0（电话）/P1（短信）/P2（邮件）/P3（站内消息）
  • 通知人员：可配置多个接收人
  • 告警收敛：相同告警在设定时间内不重复发送

6.4 日志查询

点击「运维管理」→「日志查询」：

  • 按技能筛选：选择特定技能查看日志
  • 按时间筛选：自定义时间范围
  • 日志级别：DEBUG/INFO/WARN/ERROR
  • 关键字搜索：支持正则表达式搜索
  • 日志导出：支持导出到文件
⚠️ 注意：生产环境日志默认保留 7 天，如需更长时间保留，请提前申请日志存储扩容。

🎯

9. 常见问题与解决方案

Q1: 需求拆解时需要注意什么？

需求拆解是将大需求分解为可执行的小需求的过程，注意事项包括：

  • 原子性：每个子需求应该是原子性的，不可再分
  • 可测试：每个子需求都应该有明确的验收标准
  • 可追溯：子需求必须能关联到具体的技能实现
  • 时间合理：单个子需求的开发周期不宜超过2周

如果需求拆解遇到困难，建议联系产品负责人协助分析。

Q2: 代码生成失败怎么办？

代码生成可能因多种原因失败，排查步骤如下：

  1. 检查输入描述：确认需求描述是否清晰、完整
  2. 查看错误日志：在代码编辑页面查看具体的错误信息
  3. 分步生成：尝试分段生成代码，先基础框架后具体实现
  4. 手动补充：AI 生成的基础上手动补充和修正
  5. 寻求帮助：在技术交流群咨询或提交工单

建议在代码生成前，详细描述业务逻辑和技术要求，可提高生成质量。

Q3: 测试用例设计有哪些最佳实践？

高质量的测试用例是保障技能质量的关键：

  • 覆盖正向流程：验证正常业务场景能正确执行
  • 覆盖异常流程：验证异常输入能被正确处理
  • 边界条件测试：测试输入的最大值、最小值、空值等边界
  • 性能测试：验证响应时间符合要求
  • 安全测试：验证敏感数据得到正确保护

测试用例��盖��应达到 90% 以上，建议使用自动化测试提高效率。

Q4: 部署失败常见原因有哪些？

部署失败时，常见的错误原因包括：

错误类型原因解决方案
构建失败代码编译错误、依赖缺失检查代码和依赖配置
镜像拉取失败网络问题或镜像仓库权限检查网络和仓库权限
健康检查失败服务启动超时或配置错误检查服务配置和日志
资源不足集群资源不足申请扩容或等待资源释放
配置校验失败环境变量或配置项错误检查环境变量配置

遇到部署失败时，首先查看部署日志中的详细错误信息。

Q5: 技能准确率下降如何分析和解决？

技能准确率下降可能有多种原因，按以下步骤排查：

  1. 确认基准：查看是否修改过代码、模型或配置
  2. 分析日志：查看失败案例的特征和模式
  3. 检查数据：确认训练数据是否被污染或过期
  4. 模型评估：重新评估模型在测试集上的表现
  5. 输入分析，分析用户输入是否与训练集差异过大

常用解决方案：补充训练数据、调整提示词、重新训练模型、增加拒识机���等。
