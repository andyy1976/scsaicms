2. 完整配置脚本

以下是完整的 .gitlab-ci.yml 配置文件，包含所有9个阶段的详细配置：

# ============================================================
# SCSAI GitLab CI/CD 流水线配置
# 版本: V2.0 | 更新日期: 2026-05-10
# ============================================================

stages:
  - code-commit
  - code-quality
  - unit-test
  - build-image
  - deploy-test
  - integration-test
  - acceptance-test
  - deploy-prod
  - update-trace

# 全局变量
variables:
  DOCKER_REGISTRY: "registry.scshb.cn"
  IMAGE_NAME: "scsai-app"
  TEST_NAMESPACE: "scsai-test"
  PROD_NAMESPACE: "scsai-prod"
  ARAS_API_URL: "https://aras.scshb.cn/api/v2"

# 阶段1: 代码提交检测
stage_commit:
  stage: code-commit
  script:
    - echo "代码已提交: $COMMIT_SHA"
    - echo "提交者: $GIT_USER_NAME"
    - echo "提交信息: $COMMIT_MESSAGE"
  only:
    - branches
  tags:
    - docker

# 阶段2: 代码质量检查
code_quality:
  stage: code-quality
  image: node:18-alpine
  script:
    - npm install -g eslint
    - eslint --version
    - eslint src/ --format stylish || true
  artifacts:
    reports:
      codequality: gl-code-quality-report.json
  only:
    - branches
  tags:
    - docker

# 阶段3: 单���测���
unit_test:
  stage: unit-test
  image: node:18-alpine
  script:
    - npm install
    - npm test -- --coverage
  coverage: '/Lines\s+:\s+(\d+\.\d+)%/'
  artifacts:
    paths:
      - coverage/
    reports:
      junit: test-results.xml
  only:
    - branches
  tags:
    - docker

# 阶段4: Docker镜像构建
build_image:
  stage: build-image
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker login -u $DOCKER_REGISTRY_USER -p $DOCKER_REGISTRY_PASS $DOCKER_REGISTRY
    - docker build -t $IMAGE_NAME:$CI_COMMIT_SHA .
    - docker tag $IMAGE_NAME:$CI_COMMIT_SHA $DOCKER_REGISTRY/$IMAGE_NAME:latest
    - docker push $DOCKER_REGISTRY/$IMAGE_NAME:$CI_COMMIT_SHA
    - docker push $DOCKER_REGISTRY/$IMAGE_NAME:latest
  only:
    - main
    - develop
  tags:
    - docker

# 阶段5: 部署到测试环境
deploy_test:
  stage: deploy-test
  image: bitnami/kubectl:latest
  script:
    - kubectl set image deployment/scsai-app scsai=$DOCKER_REGISTRY/$IMAGE_NAME:$CI_COMMIT_SHA -n $TEST_NAMESPACE
    - kubectl rollout status deployment/scsai-app -n $TEST_NAMESPACE --timeout=300s
  environment:
    name: test
    url: https://test.scshb.cn
  only:
    - develop
  tags:
    - kubernetes

# 阶段6: 集成测试
integration_test:
  stage: integration-test
  image: postman/newman:alpine
  script
    - newman run postman/collections/*.json --environment postman/envs/test.json --reporters cli,json
  artifacts:
    paths:
      - newman/
  needs:
    - deploy_test
  only:
    - develop
  tags:
    - docker

# 阶段7: 验收测试（手动触发）
acceptance_test:
  stage: acceptance-test
  script:
    - echo "等待业务方验收确认..."
    - echo "验收通过后继续流水线"
  when: manual
  needs:
    - integration_test
  only:
    - develop

# 阶段8: 部署到生产环境
deploy_prod:
  stage: deploy-prod
  image: bitnami/kubectl:latest
  script:
    - kubectl set image deployment/scsai-app scsai=$DOCKER_REGISTRY/$IMAGE_NAME:$CI_COMMIT_SHA -n $PROD_NAMESPACE
    - kubectl rollout status deployment/scsai-app -n $PROD_NAMESPACE --timeout=600s
  environment:
    name: production
    url: https://prod.scshb.cn
  when: manual
  needs:
    - acceptance_test
  only:
    - main
  tags:
    - kubernetes

# 阶段9: 更新ARAS追溯表
update_aras_trace:
  stage: update-trace
  image: alpine:latest
  script:
    - apk add --no-cache curl jq
    - |
      curl -X POST $ARAS_API_URL/trace/update \
        -H "Content-Type: application/json" \
        -H "Authorization: Bearer $ARAS_API_TOKEN" \
        -d '{
          "version": "'$CI_COMMIT_SHA'",
          "commit": "'$CI_COMMIT_SHA'",
          "branch": "'$CI_COMMIT_BRANCH'",
          "author": "'$GIT_USER_EMAIL'",
          "timestamp": "'$(date -u +%Y-%m-%dT%H:%M:%SZ)'",
          "environment": "production"
        }'
  needs:
    - deploy_prod
  only:
    - main
  tags:
    - docker

3. 环境变量配置说明

在 GitLab 项目设置 → CI/CD → 变量 中配置以下敏感信息：

变量名 说明 是否必须 示例值
ARAS_USERNAME ARAS系统用户名 是 scsai_admin
ARAS_PASSWORD ARAS系统密码（需Protect） 是 ********
ARAS_API_TOKEN ARAS API访问令牌（需Protect） 是 aras_xxxxxxxxx
DOCKER_REGISTRY_USER 镜像仓库用户名 是 scsai_ci
DOCKER_REGISTRY_PASS 镜像仓库密码（需Protect） 是 ********
PLATFORM_API_TOKEN 平台API调用令牌 是 platform_xxxxxxxxx
KUBECONFIG Kubernetes配置（需Protect） 是 base64编码内容
SONAR_TOKEN SonarQube项目Token 否 xxxxxxxxxxxx
⚠️ 安全建议：
  • 所有包含密码/Token的变量必须勾选 Protect variable
  • 不推荐在流水线日志中输出敏感变量
  • 定期轮换Token，建议90天更换一次

4. 部署与配置步骤

流水线部署需要按以下6个步骤完成：

步骤1: 准备Kubernetes环境

在目标集群创建所需的namespace和ServiceAccount：

# 创建命名空间
kubectl create namespace scsai-test
kubectl create namespace scsai-prod

# 创建ServiceAccount
kubectl create serviceaccount gitlab-runner -n scsai-test
kubectl create serviceaccount gitlab-runner -n scsai-prod

# 绑定RBAC权限
kubectl bind clusterrolebinding gitlab-runner-test --clusterrole=edit --serviceaccount=scsai-test:gitlab-runner
步骤2: 配置GitLab Runner

在K8s集群中安装并注册GitLab Runner：

# 添加GitLab官方Chart仓库
helm repo add gitlab https://charts.gitlab.io

# 安装Runner
helm install gitlab-runner gitlab/gitlab-runner \
  --set gitlabUrl=https://gitlab.scshb.cn \
  --set runnerRegistrationToken=<TOKEN> \
  --set rbac.create=true \
  --namespace gitlab-runner
步骤3: 配置镜像仓库

配置私有Docker Registry并设置认证：

# 创建Registry Secret
kubectl create secret docker-registry registry-secret \
  --docker-server=registry.scshb.cn \
  --docker-username=$DOCKER_REGISTRY_USER \
  --docker-password=$DOCKER_REGISTRY_PASS \
  --namespace scsai-test
步骤4: 配置CI/CD变量

在GitLab项目设置中配置所有必需变量：

  • 进入项目 → 设置 → CI/CD → 变量
  • 逐一添加环境变量，勾选"Protect variable"
步骤5: 创建部署清单

在项目中创建Kubernetes部署配置文件：

apiVersion: apps/v1
kind: Deployment
metadata:
  name: scsai-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: scsai-app
  template:
    spec:
      containers:
      - name: scsai
        image: registry.scshb.cn/scsai-app:latest
        ports:
        - containerPort: 8080
步骤6: 验证流水线

推送代码到develop分支触发流水线，验证各阶段执行：

  • 检查代码质量阶段是否通过
  • 确认单元测试覆盖率达标
  • 验证测试环境部署成功

5. 流水线触发方式

流水线支持三种触发方式：

触发方式 配置 使用场景
自动触发 only: - branches 代码推送到指定分支自动执行
手动触发 when: manual 需要人工确认的关键节点
API触发 GitLab API 外部系统集成调用

5.1 API触发方式

使用GitLab API触发流水线的示例：

# 触发指定分支的流水线
curl -X POST \
  "https://gitlab.scshb.cn/api/v4/projects/123/pipeline" \
  -H "PRIVATE-TOKEN: <GITLAB_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "ref": "develop",
    "variables": [
      {"key": "DEPLOY_ENV", "value": "test"}
    ]
  }'
💡 提示：手动触发阶段（如验收测试、生产部署）需要在GitLab UI中点击"Play"按钮确认执行。

6. 最佳实践

6.1 灰度发布策略

生产环境采用灰度发布策略，减少升级风险：

  • 初始流量：10%新版本 → 90%旧版本
  • 观察期：30分钟无错误则增加到50%
  • 完全切换：确认无异常后全量更新
# 金丝雀发布示例
kubectl patch deployment scsai-app -n scsai-prod \
  -p '{"spec":{"replicas": 3}}'

# 新版本增加1个实例
kubectl set image deployment/scsai-app scsai=$IMAGE_NAME:$NEW_VERSION -n scsai-prod

# 观察完成后全量更新
kubectl rollout undo deployment/scsai-app -n scsai-prod

6.2 版本管理规范

  • 版本号格式：vMAJOR.MINOR.PATCH（如v1.2.0）
  • 主版本号：不兼容的API变更
  • 次版本号：向后兼容的功能新增
  • 修订号：向后兼容的缺陷修复

6.3 安全最佳实践

  • 镜像扫描：使用Trivy扫描CVE漏洞
  • 密钥管理：使用Vault存储敏感信息
  • 网络策略：配置K8s网络策略限制访问
  • 审计日志：记录所有部署操作

6.4 监控与告警

7. 常见问题与解决方案

Q1: 流水线卡在"等待手动触发"阶段？
解决方案：该阶段需要人工点击"Play"按钮确认。在流水线页面找到该任务，点击右侧行动按钮 → 继续操作。
Q2: Docker镜像构建失败，提示"denied: requested access to the resource is denied"？
解决方案：检查DOCKER_REGISTRY_USER和DOCKER_REGISTRY_PASS变量是否正确，以及是否有该仓库的推送权限。
Q3: 部署到测试环境失败，Pod一直处于Pending状态？
解决方案：执行kubectl describe pod <pod-name> -n scsai-test查看原因，常见原因：资源不足、存储卷未挂载成功。
Q4: ARAS追溯表更新失败？
解决方案：验证ARAS_API_TOKEN是否正确，以及API endpoint是否可达。可在本地先测试API连通性。
Q5: 如何回滚到之前版本？
解决方案：在GitLab流水线页面，找到成功的历史运行记录，点击对应版本的"部署生产"阶段手动执行；或在K8s中使用kubectl rollout undo deployment/scsai-app -n scsai-prod回滚。
Q6: 单元测试覆盖率不达标？
解决方案：在项目根目录配置jest.config.js，设置覆盖率阈值。确保新增代码有对应的测试用例。
Q7: GitLab Runner内存不足导致任务失败？
解决方案：在Runner配置中增加资源限制，或使用更大的Runner实例类型。
Q8: 如何跳过某些阶段直接部署？
解决方案：在GitLab CI/CD设置中，可以手动选择跳过某些阶段，但需要注意依赖关系可能导致任务无法执行。
